### Сценарий: интернет пропал.

На планшете (Web POS): при попытке открыть страницу браузер покажет ошибку сети. Однако можно заранее загрузить offline-версию PWA (Progressive Web App) – кэшировать HTML/JS/CSS. При разрыве связи кассир переходит в локальный режим (определяется автоматически).

Edge Gateway продолжает работать локально. Планшет отправляет запросы на Gateway по локальной сети (Wi-Fi без интернета). Gateway:

Сверяет цены с локальным кэшем (копия справочника, полученная последний раз из облака).

Принимает оплату наличными (безналичная оплата в офлайн невозможна, можно только наличные или отложить до восстановления связи).

Формирует фискальный чек (фискальный регистратор работает локально).

Сохраняет данные о чеке в локальное хранилище SQLite (или Redis) в виде очереди событий.

После восстановления интернета:

Edge Gateway устанавливает соединение с Cloud POS Backend.

Gateway отправляет все несинхронизированные чеки в облако (POST /sync/transactions).

Облако проверяет их (например, на дублирование по уникальному ID чека), обновляет остатки, финансовые данные.

При необходимости Gateway получает обновлённый справочник товаров и цен.

Планшет (PWA) обновляет кэш.

Детали механизма синхронизации:

Идемпотентность: каждый чек имеет уникальный receipt_id, сгенерированный локально (UUID + timestamp). При повторной отправке облако игнорирует дубли.

Очередь с retry: если отправка не удалась (таймаут, ошибка 5xx), Gateway повторяет попытку с экспоненциальной задержкой (10 с, 30 с, 1 мин, 5 мин).

Конфликты: если во время офлайн-периода изменились цены (например, в облаке запустили акцию), то после синхронизации локальный кэш обновляется, но уже проданные товары фиксируются по старым ценам – это допустимо (магазин не теряет деньги, возможна небольшая погрешность, покрывается из резерва).

Мониторинг: облако логирует все отложенные транзакции и может уведомить администратора, если какой-то магазин долго не выходит на связь.
